# Tezza Lending Application

Spring Boot lending case study implementing product configuration, customer profiles, loan disbursement, repayments, overdue sweeps, and event-driven notifications.

## Requirements

- JDK 17+
- Maven 3.9+

## Steps to Run

1. Clone the repository and enter the project folder:

```bash
git clone <repository-url>
cd Loans
```

2. Run the tests:

```bash
mvn test
```

3. Start the application with the default file-based H2 database:

```bash
mvn spring-boot:run
```

The service starts on `http://localhost:8080`. Use these links after startup:

- Swagger UI: `http://localhost:8080/swagger-ui.html`
- OpenAPI JSON: `http://localhost:8080/v3/api-docs`
- Committed OpenAPI YAML: [`docs/openapi.yaml`](docs/openapi.yaml)
- Static Swagger viewer for GitHub Pages: [`docs/swagger.html`](docs/swagger.html)
- H2 console: `http://localhost:8080/h2-console`

When the repository is uploaded to GitHub, enable GitHub Pages from the `docs`
folder to view the committed Swagger UI page. The page reads `docs/openapi.yaml`,
so it can show the API contract without running the backend.

## Database Profiles

The default profile is `h2-file`, which stores data under `./data/lending-db`.
Use these values for the H2 console:

- JDBC URL: `jdbc:h2:file:./data/lending-db`
- Username: `sa`
- Password: leave blank

Run with in-memory H2:

```bash
SPRING_PROFILES_ACTIVE=h2-memory mvn spring-boot:run
```

Run with PostgreSQL:

```bash
export SPRING_PROFILES_ACTIVE=postgres
export POSTGRES_JDBC_URL="jdbc:postgresql://localhost:5432/tezza_lending"
export POSTGRES_USERNAME="password"
export POSTGRES_PASSWORD="password"

mvn spring-boot:run
```

Optional H2 overrides:

```bash
export H2_FILE_PATH="./data/lending-db"
export H2_DATABASE_NAME="lending-db"
export H2_USERNAME="sa"
export H2_PASSWORD=""
```

## Gmail Notifications

Email notifications use Gmail SMTP through Spring Mail. Gmail requires an app password, not your normal Gmail password.

Set these environment variables before starting the app:

```bash
export GMAIL_USERNAME="your-gmail-address@gmail.com"
export GMAIL_APP_PASSWORD="your-16-character-app-password"
export GMAIL_FROM="your-gmail-address@gmail.com"
```

When a notification resolves to the `EMAIL` channel, the application sends the rendered template to the customer's email address through Gmail. Gmail notifications are enabled by default; set `GMAIL_NOTIFICATIONS_ENABLED=false` only when you intentionally want to disable delivery. If Gmail delivery is disabled or misconfigured, the notification is stored with `FAILED` status and a `deliveryDetail` explaining why.

Customer registration always creates a 6-digit OTP and sends a styled welcome email through Gmail. Customer numbers are generated by the system in the format `CUST-yyyyMMdd-####`.

## Case Study Coverage

- Product configuration supports tenure in `DAYS` or `MONTHS`, service fees, daily accrual fees, late fees, and fee trigger days.
- Loan management supports lump-sum and installment structures, individual due dates, optional consolidated due dates, repayment, cancellation, write-off, overdue sweeps, and due-date reminder sweeps.
- Customer profiles store personal/contact details, risk grade, preferred notification channel, generated customer numbers, registration OTPs, and adjustable loan limits.
- Notifications are event-driven for customer registration, loan disbursement, due-date reminders, repayments, full repayment, and overdue notices.
- Notification rules can route by event type, loan product, customer segment, and customer channel preference.
- Notification templates are stored in the database and support variable substitution.
- Flyway creates the database schema and loads seed data automatically.
- Unit tests cover key loan and notification behavior.

## Architecture

The code is organized by domain module:

- `product`: loan product setup, tenure, and fee configuration.
- `customer`: customer profile and loan limit management.
- `loan`: disbursement, installments, repayment, loan state transitions, and overdue sweep jobs.
- `notification`: templates, rules, event-driven notification generation.
- `logging`: request/response logging with request ID correlation.

Services are exposed through interfaces and implemented with Spring-managed classes. Controllers only handle REST concerns and delegate business behavior to services. Persistence is handled through Spring Data JPA repositories. Flyway creates the schema and loads seed data automatically at startup.

## Logging Strategy

Every HTTP request is logged with this format:

```text
SourceSystem ={} || TargetSystem={} || RequestId={} || RequestTime={} || RequestPayload={} || ResponsePayload={} || Method={} || Path={} || Status={}
```

Clients can send `X-Request-Id`; otherwise the API generates one and returns it in the response header.

## Seed Data

Flyway seeds:

- Products: `PAYDAY-30`, `SME-12M`
- Customers: `CUST-001`, `CUST-002`
- Product fees: service fee, daily fee, and late fee examples
- Notification templates and routing rules

## Sample API Calls

Create a product:

```bash
curl -X POST http://localhost:8080/api/v1/products \
  -H 'Content-Type: application/json' \
  -H 'X-Request-Id: demo-product-001' \
  -d '{
    "code": "MICRO-14",
    "name": "14 Day Micro Loan",
    "minimumAmount": 500,
    "maximumAmount": 20000,
    "tenureValue": 14,
    "tenureUnit": "DAYS",
    "daysAfterDueForFeeApplication": 2,
    "active": true,
    "fees": [
      {
        "feeType": "SERVICE_FEE",
        "calculationType": "PERCENTAGE",
        "applicationStage": "ORIGINATION",
        "value": 4.5
      },
      {
        "feeType": "LATE_FEE",
        "calculationType": "FIXED",
        "applicationStage": "AFTER_DUE_DATE",
        "value": 500,
        "triggerDaysAfterDue": 2
      }
    ]
  }'
```

Create a customer:

```bash
curl -X POST http://localhost:8080/api/v1/customers \
  -H 'Content-Type: application/json' \
  -d '{
    "firstName": "Grace",
    "lastName": "Njeri",
    "email": "grace.njeri@example.com",
    "phoneNumber": "+254700000003",
    "dateOfBirth": "1990-06-20"
  }'
```

The backend generates `customerNumber` and registration OTP. It also defaults
`preferredNotificationChannel` to `EMAIL`, `riskGrade` to `UNRATED`, and
`loanLimit` to `0.00` until the limit is assigned through the limit endpoint.

Disburse an installment loan using seeded records:

```bash
curl -X POST http://localhost:8080/api/v1/loans \
  -H 'Content-Type: application/json' \
  -d '{
    "customerId": 1,
    "productId": 1,
    "principalAmount": 10000,
    "installmentCount": 4,
    "consolidatedBilling": false
  }'
```

When `structure` is not supplied, the backend defaults it to `LUMP_SUM`. Send
`"structure": "INSTALLMENT"` only when the user explicitly chooses installments.

Record repayment:

```bash
curl -X POST http://localhost:8080/api/v1/loans/1/repayments \
  -H 'Content-Type: application/json' \
  -d '{
    "amount": 2500
  }'
```

The backend defaults repayment `channel` to `SYSTEM` and generates
`externalReference` when the payment provider has not supplied one.

Run overdue sweep manually:

```bash
curl -X POST 'http://localhost:8080/api/v1/loans/sweeps/overdue?businessDate=2026-07-30'
```

Send due-date reminders manually:

```bash
curl -X POST 'http://localhost:8080/api/v1/loans/sweeps/due-reminders?businessDate=2026-07-27&daysAhead=3'
```

List generated notifications:

```bash
curl http://localhost:8080/api/v1/notifications
```

## Loan States

Loans move through:

- `OPEN`: active and in good standing.
- `OVERDUE`: due date has passed and the overdue sweep has processed it.
- `CLOSED`: outstanding balance is fully repaid.
- `CANCELLED`: cancelled before completion.
- `WRITTEN_OFF`: marked uncollectable.
